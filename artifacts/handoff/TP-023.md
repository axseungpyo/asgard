## Objective

도메인 이벤트와 EventBus를 도입하여 Watcher, WebSocket, Use Case 간의 결합도를 이벤트 기반으로 해소한다.

## Scope In

### 1. core/ports/IEventBus.ts — 이벤트 버스 인터페이스
```typescript
interface DomainEvent {
  type: string;
  timestamp: number;
  payload: Record<string, unknown>;
}

interface IEventBus {
  publish(event: DomainEvent): void;
  subscribe(eventType: string, handler: (event: DomainEvent) => void): () => void;
  subscribeAll(handler: (event: DomainEvent) => void): () => void;
}
```

### 2. core/events/ — 도메인 이벤트 정의
- `DomainEvent.ts` — 기본 이벤트 타입
- `AgentStarted.ts` — { agent, tp, mode, pid }
- `AgentStopped.ts` — { agent, exitCode }
- `TaskCreated.ts` — { id, title, agent }
- `TaskStatusChanged.ts` — { id, oldStatus, newStatus }
- `CommandProcessed.ts` — { command, skill, result }
- `FileChanged.ts` — { path, type: "log" | "index" | "handoff" }
- `LogEntry.ts` — { agent, message, level }

### 3. adapters/events/InProcessEventBus.ts — 구현체
- Node.js EventEmitter 기반
- Observer 패턴
- 동기 이벤트 발행 (비동기 핸들러는 fire-and-forget)

### 4. Watcher → EventBus 연결
- `infra/watcher.ts` 수정: 기존 직접 콜백 대신 EventBus.publish() 사용
- 파일 변경 감지 시 FileChanged 이벤트 발행
- 로그 변경 시 LogEntry 이벤트 발행

### 5. WebSocket → EventBus 구독
- `websocket/index.ts` 수정: EventBus.subscribe()로 이벤트 수신
- logs-handler: LogEntry 이벤트 → 클라이언트 전송
- status-handler: TaskStatusChanged, AgentStarted/Stopped → 상태 갱신 후 전송
- odin-handler: CommandProcessed → 클라이언트 전송

### 6. Use Case에서 이벤트 발행
- StartAgentUseCase → AgentStarted 이벤트 발행
- StopAgentUseCase → AgentStopped 이벤트 발행
- CreateTaskUseCase → TaskCreated 이벤트 발행
- UpdateTaskStatusUseCase → TaskStatusChanged 이벤트 발행
- ProcessCommandUseCase → CommandProcessed 이벤트 발행

### 7. di/container.ts 확장
- EventBus 인스턴스 생성 (싱글톤)
- Use Case에 EventBus 주입
- Watcher에 EventBus 주입
- WebSocket setup에 EventBus 전달

## Scope Out
- 이벤트 영속화 (이벤트 소싱) — 현재 규모에서 불필요
- 비동기 메시지 큐 (RabbitMQ 등) — 단일 프로세스이므로 in-process 충분
- 프론트엔드 변경 없음 (WebSocket 메시지 포맷 동일 유지)

## Implementation Notes

### 이벤트 흐름 (After)
```
파일 변경 → Watcher → EventBus.publish(FileChanged)
                              ↓
                     EventBus subscribers:
                       ├→ WebSocket status-handler → 클라이언트
                       ├→ WebSocket logs-handler → 클라이언트
                       └→ (향후) MetricsCollector, AuditLogger 등

Use Case 실행 → EventBus.publish(AgentStarted)
                              ↓
                     EventBus subscribers:
                       ├→ WebSocket odin-handler → 클라이언트
                       └→ (향후) NotificationService 등
```

### 주의사항
- 기존 Watcher의 EventEmitter를 제거하지 말고, EventBus로 브릿지하는 방식으로 점진적 전환
- WebSocket 핸들러의 기존 동작(초기 상태 전송 등)은 유지
- 이벤트 핸들러에서 에러가 발생해도 다른 핸들러에 영향 없어야 함 (에러 격리)
